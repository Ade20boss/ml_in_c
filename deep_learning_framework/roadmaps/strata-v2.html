Wire Protocol — The Memory Contract

JSON is too slow for moving gigabytes of multi-dimensional vector arrays. Communication between Habeeb's AI pipeline and the C database uses a strict binary protocol. All vectors originate as exactly 384 dimensions (standard all-MiniLM-L6-v2 output) — but past Phase 3, most of what crosses the wire is compressed.

Cache alignment is critical

VectorPayload is padded to exactly 1568 bytes and PQCodePayload to exactly 64 bytes — both clean multiples of the 32-byte cache line. This is not style; it is a performance invariant that must be preserved across every protocol change, including the new opcodes.

02

Core Engine Subsystems — Daniel's Domain

The engine is a graph structure in memory that traverses millions of nodes without scanning the entire dataset. v2.0 adds a compression layer beneath that graph, a concurrency model that lets it absorb live writes, and a crash-recovery contract that survives a kill -9 mid-stream.

Subsystem A
Memory-Mapped Vault & OS-Level Tuning

A massive contiguous file is mmap()'d into virtual memory. One million 384D vectors consume roughly 1.5 GB of RAM for raw arrays alone — before accounting for graph link metadata. Memory mapping lets the Linux kernel automatically page data in and out of physical RAM, enabling datasets that exceed server memory limits without manual disk I/O code.

Kernel-managed paging

The kernel treats the mmap region as demand-paged virtual memory. Pages not recently accessed are swapped to disk transparently. The database code reads and writes as if all data is in RAM.

Two tuning passes harden this for HNSW's access pattern, which is the opposite of sequential:

madvise(..., MADV_RANDOM). Called immediately after mapping, this disables the kernel's sequential read-ahead. HNSW hops between distant graph nodes — read-ahead would waste bandwidth pulling in pages the traversal will never touch.
Huge Pages (MAP_HUGETLB). Configuring 2MB or 1GB huge pages collapses the number of page-table entries the TLB has to track. During a graph traversal that touches thousands of scattered 4KB pages, this is the difference between a TLB hit and a TLB miss on nearly every hop.
Subsystem B New
Product Quantization & Asymmetric Distance Computation

Storing 1.5 GB per million vectors doesn't scale to the dataset sizes this engine is meant for. Product Quantization (PQ) compresses each 384D vector into a 48-byte code by splitting it into 48 sub-vectors of 8 dimensions, then replacing each sub-vector with the ID of its nearest centroid in a codebook Habeeb trains offline via K-means.

The engine loads the codebook (48 sub-spaces × 256 centroids × 8 floats) once at boot. At query time, it cannot simply decompress and run the standard L2 kernel — that would defeat the point. Instead it runs Asymmetric Distance Computation (ADC):

dADC(q, code) = Σm=148 table[m][code[m]] Sum of 48 precomputed lookup-table entries — one per sub-space

For each incoming query, the engine first computes the distance from the raw query vector to all 256 centroids in each of the 48 sub-spaces, storing the results in an L1-cached lookup table. From there, scoring any compressed candidate is just 48 table reads and a sum — accelerated with AVX-2 gather instructions to pull multiple table entries per cycle.

32× memory reduction. 1536 bytes of floats become 48 bytes of codes. One million vectors drops from ~1.5 GB to ~48 MB of hot data — small enough to live almost entirely in cache during traversal.
Subsystem C
Hierarchical Navigable Small World Graph

HNSW is a multi-layered skip-list applied to a proximity graph. Each layer contains exponentially fewer nodes than the one below it. Search begins at the sparsest top layer, taking "highway" jumps across the vector space, then descends for precise local search in Layer 0. Every node now carries a pointer to its 48-byte PQCodePayload instead of a raw float array.

Search complexity is O(log N) versus the O(N) of brute-force scan. At one million vectors, that is the difference between roughly 20 hops and one million comparisons — with every hop now scored by the ADC kernel from Subsystem B instead of a full 384D Euclidean distance.

Subsystem D New
Concurrency & The Synchronization Lock

Once ingestion is asynchronous, two things can happen to the same graph node at once: a background worker is writing a freshly-inserted vector's links while a search thread is reading them to decide its next hop. Without protection, that's a corrupted traversal or a crash.

The fix is a lightweight read-write lock (or, where contention allows, a lock-free atomic pointer swap) scoped to individual graph nodes rather than the whole graph — a global lock would serialize every search behind every insert, which defeats the point of an asynchronous pipeline.

Zero-Leak Audit. The full engine runs under Valgrind's memcheck (uninitialized memory) and massif (heap profiling) until every uninitialized-read and graph-pointer leak is resolved. A vector database that's fast but leaks memory under live writes just fails more slowly.
Subsystem E New
Fault Tolerance & The High-Water Mark

A vector database that loses writes on crash isn't production-grade. The goal is zero data loss without paying for a Write-Ahead Log on every insert — instead, the engine checkpoints in batches and exposes exactly how far it got.

Every 1,000 vectors processed, the engine calls msync(..., MS_SYNC) on the mmap vault, forcing the kernel to flush those pages to physical disk — then writes an 8-byte last_safely_synced_doc_id into a tiny adjacent .meta file.

On reconnect, the Python pipeline needs to know exactly where the C engine left off. That's the offset handshake — opcode 0x03 ties the C-side high-water mark to Habeeb's resume logic:

1 Python

Daemon boots, or reconnects after a dropped TCP connection.

2 Python

Sends opcode 0x03 — Query Sync State — before streaming any vectors.

3 C Engine

Reads the 8-byte last_safely_synced_doc_id from the .meta file and returns it as a SyncStateResponse.

4 Python

Drops every vector prior to that ID from its local holding deque, then resumes streaming from the exact offset the C engine left off at.

This is the seam the whole project is built on: at the end of every 2-week phase, Habeeb's Python output plugs into Daniel's C engine without either side guessing about state.

03

Semantic Pipeline — Habeeb's Domain

An empty vector database returns nothing. The ingestion pipeline converts human language into the 384-dimensional float arrays the engine operates on — then, in v2.0, compresses them, makes them retrievable as text again, and keeps the stream alive through crashes and reconnects.

Subsystem A
Ingestion & Chunking Engine

A Python daemon using sentence-transformers downloads a large technical corpus (target: 50,000 Wikipedia medical articles), chunks the text into clean ~512-token paragraphs without destroying semantic meaning, and passes each chunk through the all-MiniLM-L6-v2 model to produce 384D float arrays.

Each array is packed via Python's struct module into the VectorPayload binary format and streamed over a TCP socket to the C database at 1,000+ vectors per second.

The model output is deterministic: identical input text always produces identical float arrays. This makes test data reproducible and pipeline debugging tractable.
Subsystem B New
Codebook Generation — K-Means Clustering

Before the C engine can compress anything, Habeeb has to teach it how. A representative 10% sample of the offline dataset is extracted, the 384-dimensional space is split into 48 sub-spaces of 8 dimensions each, and K-means runs independently on each sub-space to produce 256 centroids — the codebook.

The codebook is serialized into a binary blob and shipped to Daniel's engine to be loaded at boot, becoming the shared vocabulary both the compressor and the ADC kernel speak.

Subsystem C New
The Quantizer — Vector Compression

For every new embedding, the quantizer splits the raw 384D float vector into 48 sub-vectors and finds the nearest centroid in the codebook for each — replacing 1536 bytes of floats with 48 bytes of 8-bit centroid IDs, packed into a PQCodePayload and streamed via opcode 0x04.

NumPy, not Python loops

A per-vector, per-sub-space Python loop chokes at this volume. The quantizer uses pure C-backed NumPy advanced indexing and broadcasting to find nearest centroids across the whole batch at once.

A verification script computes the approximate ADC distance between a query and a compressed vector in Python — this becomes the gold standard the C engine's ADC kernel is checked against.

Subsystem D New
Reverse-Lookup API & Document Management

The C engine only knows doc_ids. A FastAPI service, built around the async def concurrency model, fronts a Redis or SQLite store mapping each doc_id to its original text chunk and source metadata.

When the WebGL terminal receives a nearest-neighbor doc_id from the C engine, it hits this API for the actual paragraph to display — closing the loop from compressed integer back to human-readable text in sub-millisecond time.

Subsystem E New
Concept Search — Query Expansion

Users type vague terms. Before embedding a query, the API can route it through a fast local LLM (e.g. Llama-3) that expands an ambiguous phrase like "cardiovascular failure" into more specific related terms, then embeds the expanded text. The vectors that reach the C engine are sharper than what the user actually typed.

Subsystem F New
Stream Hardening & The Resume Logic

The TCP client talking to the C engine implements exponential backoff for connection drops and proper handling of EPIPE (broken pipe) errors. The Avalanche Test proves it: stream 10,000 vectors over the live socket as fast as the network allows, with zero struct tearing on the C side.

On every boot or reconnect, the daemon sends opcode 0x03 before any vectors, parses the returned last_safely_synced_doc_id, drops everything before that ID from its local holding deque, and resumes from the exact offset the C engine left off at — the Python half of the handshake described in Subsystem E of the Core Engine.

04

Strata Terminal — Usaamah's Domain

Clients do not reason in multidimensional arrays. The terminal proves the engine is thinking spatially by making vector space visible — and in v2.0, by visibly surviving the things that would normally break a live demo: clicks on a million points, a live data injection, and a dropped connection.

Subsystem A
3D Render Engine — Next.js + React Three Fiber

Rendering 100,000 individual objects with separate draw calls drops the frame rate to zero. The terminal uses THREE.InstancedMesh to render the entire dataset — every star, every data point — in a single WebGL draw call. The GPU processes instance transform matrices in parallel rather than sequentially.

1 draw call = 100,000 data points

Without instanced rendering, every sphere is a separate GPU command. With InstancedMesh the entire dataset — positions, colors, sizes — is uploaded as a single buffer, and the GPU iterates internally. This is non-negotiable at this scale.

Semantic Galaxy. The background is a live 3D scatter plot where vectors with high cosine similarity cluster visually near each other — Habeeb's PCA/t-SNE reduction of the 384D space to 3D coordinates, mapped into a single Float32Array and pushed directly into VRAM.
Subsystem B New
GPU Color-Picking & Spatial Navigation

Letting the user fly through the galaxy and click individual points is where naive implementations die: standard raycasting means the CPU checks the mouse line against a million points, every frame.

Instead, the terminal renders a hidden scene where every point has a unique RGB color encoding its doc_id. On click, gl.readPixels reads the single pixel under the cursor, decodes the hex back to an ID, and the terminal instantly knows which node was clicked — at zero per-frame CPU cost.

Smooth, damped camera controls (zoom, pan, rotate) sit on top of this, giving the navigation a cinematic feel rather than a CAD-tool one.
Subsystem C New
The HNSW Traversal Visualizer

Sub-millisecond logarithmic search is invisible by default — it's just a fast answer. To make it seen, the backend returns not just the final result but the array of nodes the HNSW graph hopped through to get there.

A custom fragment shader draws glowing, animated lines connecting these nodes in sequence. The user physically watches the algorithm jump from the sparse Layer 3 entry point down to the dense Layer 0 cluster — Daniel's diagram from Section 02, alive and moving.

Subsystem D New
The Data Avalanche — Live Injection

If the frontend parses thousands of incoming WebSocket packets on the main thread, the UI freezes. Socket listening and binary parsing are shifted entirely into a Web Worker, using SharedArrayBuffer to hand parsed float arrays back to the main thread without copying.

The main thread dynamically expands the InstancedMesh buffer and injects the new points — the galaxy visibly grows in real time as Habeeb's background workers stream compressed vectors in.

Subsystem E New
Terminal UI Overlay & Reverse Lookup

A command-line-style 2D interface sits over the 3D canvas. When the user types a query, it routes to Habeeb's FastAPI Query Expander, triggers the C engine search, and the camera flies to the result cluster.

When a doc_id comes back, the UI fetches the human-readable paragraph from the reverse-lookup API and displays it in a glass-morphic overlay panel — completing the loop from user thought, to visual data, to actual text.

Subsystem F New
State Reconciliation & Reconnection

If the WebSocket drops, the UI doesn't panic — it shows a subtle "Re-establishing Telemetry" indicator and reconnects using exponential backoff.

On reconnect, it queries the FastAPI layer for any doc_ids injected while it was offline, pulls the missing vectors, updates the GPU buffer, and flashes the new nodes in — ensuring the visual state mirrors the C engine's mmap vault and its last_safely_synced_doc_id exactly.
