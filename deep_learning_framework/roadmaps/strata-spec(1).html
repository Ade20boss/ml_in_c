Wire Protocol — The Memory Contract

JSON is too slow for moving gigabytes of multi-dimensional vector arrays. Communication between Habeeb's AI pipeline and the C database uses a strict binary protocol. All vectors are exactly 384 dimensions (standard all-MiniLM-L6-v2 output).

Struct 01 — Request Header
typedef struct PacketHeader __attribute__((packed)) · 12 bytes
uint32_t magic // 0xDEADBEEF — packet integrity guard
uint16_t version // protocol version
uint8_t opcode // 0x01 = Insert Batch · 0x02 = Query NN
uint8_t reserved // align header to 8 bytes
uint32_t count // number of vectors in payload
uint32_t dim // fixed to 384
Struct 02 — Inbound Vector Payload
typedef struct VectorPayload __attribute__((packed)) · 1568 bytes
uint32_t doc_id // unique document identifier
float vector[384] // 1536 bytes of contiguous float data
uint8_t padding[28] // pad to 32-byte cache-line multiple
Struct 03 — Outbound Search Result
typedef struct SearchResult __attribute__((packed)) · 8 bytes
uint32_t doc_id // ID of closest match
float distance // similarity score
Cache alignment is critical

VectorPayload is padded to exactly 1568 bytes — a perfect multiple of 32 — so every struct begins on a 32-byte cache-line boundary. This is not style; it is a performance invariant that must be preserved across all protocol changes.

02

Core Engine Subsystems — Daniel's Domain

The engine is a graph structure in memory that traverses millions of nodes without scanning the entire dataset. A naïve linear scan collapses at scale — HNSW provides logarithmic traversal cost instead.

Subsystem A
Memory-Mapped Vault

A massive contiguous file is mmap()'d into virtual memory. One million 384D vectors consume roughly 1.5 GB of RAM for raw arrays alone — before accounting for graph link metadata. Memory mapping lets the Linux kernel automatically page data in and out of physical RAM, enabling datasets that exceed server memory limits without manual disk I/O code.

Kernel-managed paging

The kernel treats the mmap region as demand-paged virtual memory. Pages not recently accessed are swapped to disk transparently. The database code reads and writes as if all data is in RAM.

Subsystem B — Hot Path
SIMD Distance Calculation

When a query vector arrives, it must be compared against every candidate vector in the search graph. The Euclidean distance kernel is the single most-executed code path in the entire system.

d(u, v) = √ Σᵢ₌₁³⁸⁴ (uᵢ − vᵢ)² Standard L2 distance across all 384 dimensions

Instead of a scalar for loop, the engine uses AVX-2 or AVX-512 intrinsics via #include <immintrin.h>. A 256-bit AVX-2 register holds 8 single-precision floats; a 512-bit AVX-512 register holds 16. The subtraction and multiplication for a full cache line of floats executes in a single clock cycle rather than 8 or 16 sequential ones.

~16× throughput improvement over scalar C on the distance kernel. This is the difference between querying 100K vectors per second and querying 1.6M vectors per second.
Subsystem C — Graph Navigation
Hierarchical Navigable Small World Graph

HNSW is a multi-layered skip-list applied to a proximity graph. Each layer contains exponentially fewer nodes than the one below it. Search begins at the sparsest top layer, taking "highway" jumps across the vector space, then descends for precise local search in Layer 0.

03

Semantic Pipeline — Habeeb's Domain

An empty vector database returns nothing. The ingestion pipeline converts human language into the 384-dimensional float arrays the engine operates on.

Subsystem A
Ingestion Engine

A Python daemon using sentence-transformers downloads a large technical corpus (target: 50,000 Wikipedia medical articles), chunks the text into paragraphs, and passes each chunk through the all-MiniLM-L6-v2 model to produce 384D float arrays.

Each array is packed into the VectorPayload binary struct format and streamed over a TCP socket to the C database at 1,000+ vectors per second.

The model output is deterministic: identical input text always produces identical float arrays. This makes test data reproducible and pipeline debugging tractable.
04

Strata Terminal — Usaamah's Domain

Clients do not reason in multidimensional arrays. The terminal proves the engine is thinking spatially by making vector space visible.

Subsystem A
3D Render Engine — Next.js + React Three Fiber

Rendering 100,000 individual objects with separate draw calls drops the frame rate to zero. The terminal uses THREE.InstancedMesh to render the entire dataset — every star, every data point — in a single WebGL draw call. The GPU processes instance transform matrices in parallel rather than sequentially.

1 draw call = 100,000 data points

Without instanced rendering, every sphere is a separate GPU command. With InstancedMesh the entire dataset — positions, colors, sizes — is uploaded as a single buffer, and the GPU iterates internally. This is non-negotiable at this scale.

Subsystem B
Visual Components
Semantic Galaxy. The background is a live 3D scatter plot where vectors with high cosine similarity cluster visually near each other. The spatial layout is the AI's semantic map made visible.
Fly-Through Camera. When a user queries, the pipeline: (1) embeds the query text, (2) retrieves the top 10 matching SearchResult IDs from the C engine, (3) animates the Three.js camera to physically traverse the galaxy and land on the corresponding cluster.
