01

Wire Protocol — The Gossip Contract

Nodes are constantly joining, leaving, and dying. Communication must be asynchronous and resilient to partial failure. UDP carries rapid peer discovery via Heartbeats; TCP carries reliable payload transfer for actual compute work.

Struct 01 — P2P Mesh Header
typedef struct MeshHeader __attribute__((packed)) · 8 bytes
uint32_t magic // 0x00BADA55 — network identifier
uint16_t version // protocol version
uint8_t msg_type // 0x01=Heartbeat · 0x02=TaskReq · 0x03=TaskResult
uint8_t ttl // time-to-live — prevents infinite network loops
Struct 02 — Heartbeat Payload (UDP · every 500ms)
typedef struct Heartbeat __attribute__((packed)) · 16 bytes
uint32_t node_id // unique cryptographic node identifier
uint32_t ip_addr // IPv4 address stored as integer
uint16_t tcp_port // port to connect for payload transfer
uint8_t cpu_load // 0–100% utilization metric
uint8_t padding[5] // align to 16 bytes
Struct 03 — Task Dispatch Payload (TCP)
typedef struct TaskPayload __attribute__((packed)) · 32 bytes
uint64_t task_id // unique computation identifier
uint32_t origin_id // node that requested the work
uint32_t payload_sz // size in bytes of the Wasm binary following this struct
uint32_t mem_req // megabytes of RAM required to execute
uint8_t padding[12] // align to 32 bytes
TTL is a hard requirement

Without TTL in the MeshHeader, a single Heartbeat re-broadcast propagates forever. Every node that receives a gossip message decrements TTL before forwarding; a packet reaching zero is silently dropped. This is what prevents the network from amplifying itself to death.

02

Core Engine Subsystems — Daniel's Domain

A Nebula node is an autonomous agent, not a web server. It must simultaneously listen for peers, accept incoming compute tasks, and dispatch outgoing work — all without a central authority telling it what to do.

Subsystem A
Gossip Protocol — Peer Discovery

There is no registry server. Nodes discover each other by broadcasting state and relaying what they hear.

1
Node A broadcasts a UDP Heartbeat to its known neighbors B and C: "I am alive. CPU load: 20%."
2
Node B receives the Heartbeat, updates its DHT entry for A, and re-broadcasts to its own neighbors D and E with ttl--.
3
Within milliseconds, the entire reachable network holds a current map of which nodes are alive and what CPU capacity each has available.
4
A node that stops broadcasting is considered dead after a configurable timeout. Any in-flight tasks assigned to it are re-queued automatically.
Subsystem B — Zero-Trust Execution
WebAssembly Sandbox Runtime

Raw C code from a remote node cannot execute directly on another machine — that would be a remote-code-execution vulnerability by definition. Every compute task is compiled to a WebAssembly binary before dispatch.

A lightweight Wasm runtime (wasm3 or wasmtime) is embedded directly into the C backend. When a TaskPayload arrives:

1
Extract the Wasm binary from the bytes following the struct header.
2
Spin up an isolated sandbox with memory capped to mem_req megabytes. The sandbox has no filesystem access and no network access.
3
Execute the compute task. Capture the return value.
4
Tear down the sandbox completely. No state persists between tasks. Send the result back as a TaskResult packet.
Sandboxing is non-negotiable

Wasm's linear memory model means the sandbox literally cannot address memory outside its allocation. A malicious task binary cannot read host process memory, access the filesystem, or make syscalls. This is containerization without Docker — built directly into the engine.

Subsystem C — State Table
Distributed Hash Table (Lock-Free)

The engine maintains an in-memory lock-free hash table of all currently active nodes, keyed by node_id. The UDP listener thread writes to this table continuously as Heartbeats arrive. The TCP dispatch thread reads from it to choose routing targets. Lock-free design prevents either thread from blocking the other.

The DHT is the engine's working memory of the network. Routing decisions, load balancing, and dead-node detection all derive from it. Staleness here propagates to every downstream system.
03

AI Orchestrator — Habeeb's Domain

In a naïve network, tasks go to the first available node. Nebula's AI acts as a distributed air traffic controller, routing work to minimize total execution time across the entire mesh.

Subsystem A
Graph Routing Engine

The routing model — a Graph Neural Network or Reinforcement Learning agent — ingests live Heartbeat data from the DHT and reasons about the topology as a whole. Two nodes competing for a task are not evaluated in isolation; the model considers the full graph of current load, latency estimates, and predicted completion times.

The routing trade-off

Node F: geographically closer (lower latency) but cpu_load at 90%. Node Z: further away but completely idle. A shortest-path algorithm picks F and queues behind its backlog. The AI model picks Z and gets the result sooner. The difference is learned from thousands of past routing decisions.

The AI achieves load balancing across a decentralized network without any master coordination server. The intelligence is distributed — every node runs its own routing model instance, fed by the same gossip data stream.
04

Nebula Terminal — Usaamah's Domain

The terminal is the mission control for a living distributed system. Its job is to make the invisible mesh visible — latency, load, and failure rendered spatially in real time.

Subsystem A
3D Mesh Topology — Next.js + Three.js

Nodes render as glowing orbs in 3D space. Edges between them represent active P2P connections. A task being dispatched from Node A to Node B produces a visible light pulse traveling along the connecting edge — the user literally watches the work move through the network.

Node color is a live readout of cpu_load from the most recent Heartbeat:

0–30% — Idle (green)
31–70% — Working (amber)
71–100% — Overloaded (red)
No heartbeat — Dead (black)
Instanced rendering is mandatory

At thousands of nodes, individual draw calls crash the frame rate. The terminal uses THREE.InstancedMesh to render the entire node set in a single GPU draw call, and a GPU particle system for the task-pulse light trails. The render cost scales with the instance count, not the node count.

Sales Mechanics — Interactive Demo Triggers

Two live interactions designed for enterprise pitches. Each proves a specific architectural claim that no slide deck can replicate.

The Chaos Monkey
A button instantly kills an active node mid-task. Its orb turns black and drops from the topology. The network detects the dropout and re-routes the task to a healthy node — without data loss.
Proves absolute fault tolerance. The client watches self-healing happen in real time. No SLA document makes this argument as convincingly.
The Auto-Scale Injection
A button spawns 50 new dummy nodes into the mesh. The AI orchestrator immediately detects the new compute capacity and migrates workload away from overloaded red nodes to idle green ones.
Proves infinite horizontal scalability. The client sees the red orbs go green in seconds. That is the cloud-native promise made tangible.
