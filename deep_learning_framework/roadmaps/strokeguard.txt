### STROKEGUARD — Master Architecture Spec v1.0

**Core Concept:** A consumer-facing, edge-compute health platform. It utilizes smartphone Photoplethysmography (PPG) and WebAssembly to extract cardiovascular pulse waves locally, streaming lightweight binary data to a centralized AI backend for sub-millisecond Atrial Fibrillation (AFib) and stroke risk prediction.

---

### 01. Wire Protocol — The Wasm-to-Backend Contract

To maintain sub-millisecond backend latency, the architecture forbids transmitting video frames. The browser transmits only the extracted, filtered numerical pulse wave over a WebSocket.

**Struct 01 — The Inbound Pulse Payload**

```c
typedef struct __attribute__((packed)) {
    uint32_t session_id;     // Unique identifier for the active user session
    uint32_t sequence;       // Packet order guard (prevents out-of-order execution)
    float pulse_wave[50];    // 50 floats representing ~1.5 seconds of extracted signal
    uint8_t padding[12];     // Pad to exactly 224 bytes (clean cache alignment)
} PulsePayload;

```

**Struct 02 — The Outbound Inference Result**

```c
typedef struct __attribute__((packed)) {
    uint32_t session_id;     // Maps back to the requesting client
    float stroke_risk;       // 0.0 to 1.0 probability of critical anomaly (AFib)
    float hrv_metric;        // Heart Rate Variability variance score
    uint32_t timestamp;      // Epoch execution time
} RiskScore;

```

---

### 02. Edge Compute & Backend Routing — Daniel's Domain

The infrastructure isolates heavy lifting to the user's device, turning the backend into a pure, high-speed traffic router.

**Subsystem A: The WebAssembly Signal Extractor**

* **The Logic:** A C program compiled to `.wasm`. It accepts a raw 1D pixel buffer (RGBA) from the browser's Canvas.
* **Spatial Averaging:** The C loop isolates the Red channel (which absorbs the most light from capillary blood flow) and computes the mean intensity of the frame.
* **Digital Signal Processing (DSP):** A Finite Impulse Response (FIR) bandpass filter written in C smooths the raw intensity data, isolating frequencies strictly between 0.5 Hz (30 BPM) and 3.0 Hz (180 BPM). Motion artifacts (huge variance spikes from finger movement) are dynamically dropped.
* **The Output:** The Wasm module hands a clean array of floats (the `PulsePayload`) back to the JavaScript wrapper.

**Subsystem B: The Telemetry Router**

* **The Ingestion Gateway:** A FastAPI asynchronous WebSocket server (or an `epoll`-backed C server) that receives the binary `PulsePayload`.
* **The Message Broker:** Incoming payloads are dumped instantly into a local Redis Stream or a lock-free C queue.
* **State Management:** The server maintains an active socket map. When the AI finishes processing, the backend routes the `RiskScore` struct directly back to the correct `session_id`.

---

### 03. Arrhythmia & Inference Pipeline — Habeeb's Domain

The intelligence layer operates entirely decoupled from network I/O, ingesting clean data and outputting clinical probabilities.

**Subsystem A: The 1D-CNN Inference Worker**

* **The Model:** A 1D Convolutional Neural Network (or LSTM) trained on clinical PPG datasets to recognize the irregular wave intervals characteristic of Atrial Fibrillation (AFib).
* **The Execution Loop:** Background Python daemon workers poll the message broker. They ingest the `pulse_wave[50]` arrays, run the forward pass, and compute the anomaly probability.
* **Vectorization:** Inference utilizes NumPy and PyTorch broadcasting to process incoming vectors without slow nested Python loops.

**Subsystem B: Baseline Calibration**

* **Dynamic Thresholding:** A user's resting heart rate varies. The AI maintains a rolling baseline for the first 10 seconds of a session, establishing the user's unique normal before flagging anomalies as critical stroke risks.

---

### 04. Clinical Terminal & Capture — Usaamah's Domain

The frontend is a standard, polished SaaS interface. It abstracts the complex Wasm and WebSockets behind a clean consumer experience.

**Subsystem A: Edge Hardware Access**

* **MediaDevices API:** Next.js components trigger the browser's camera API, specifically requesting the flashlight toggle (essential for illuminating the capillaries for finger-PPG).
* **The Canvas Proxy:** Raw video frames are drawn to a hidden `<canvas>` element at 30 or 60 FPS. The `ImageData` buffer is extracted and passed to Daniel's Wasm module via Web Workers, ensuring the main UI thread never freezes.

**Subsystem B: Real-Time Visualization**

* **The Medical Tape:** Using WebGL or highly optimized Canvas 2D, the UI plots the returned `RiskScore` and the raw pulse wave into a scrolling time-series graph, mimicking a hospital ECG monitor.
* **The Alert State:** If the WebSocket returns a `stroke_risk` above the critical threshold, the UI triggers an immediate, undeniable visual alert state (red flashing gradients, haptic feedback on mobile).
