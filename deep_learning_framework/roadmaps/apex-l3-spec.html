01

Wire Protocol — The Single Source of Truth

Before any logic is written, the engine, the bots, and the UI agree on one binary format. No JSON parsing in the critical path. Every struct uses __attribute__((packed)) to remove compiler-dependent padding, with manual padding added to hit exact 32-byte cache-line alignment.

Struct 01 — Connection Header
typedef struct PacketHeader __attribute__((packed)) · 8 bytes
uint32_t magic // 0xCAFEBABE — packet integrity guard
uint16_t version // protocol version
uint16_t count // number of Order structs following
Struct 02 — Inbound Order Payload
typedef struct Order __attribute__((packed)) · 32 bytes exactly
uint32_t id // unique order identifier
uint64_t price // fixed-point integer — $10.50 = 1050
uint64_t quantity // number of units
uint8_t side // 0 = Bid (Buy) · 1 = Ask (Sell)
uint8_t type // 0 = Limit · 1 = Market
uint8_t padding[10] // pad to 32 bytes for L1 cache alignment
Struct 03 — Outbound Execution Record
typedef struct ExecutionRecord __attribute__((packed)) · 32 bytes
uint32_t maker_id // the order resting in the book
uint32_t taker_id // the incoming order that matched
uint64_t exec_price // price at which the trade executed
uint64_t exec_qty // quantity exchanged
uint64_t timestamp // epoch time in nanoseconds
32 bytes is the contract, not a coincidence

Order and ExecutionRecord are both exactly 32 bytes — one L1 cache line. The matching thread can load a full order with a single cache fetch. Any future field addition must come out of the padding budget, not extend the struct.

02

Core Engine Subsystems — Daniel's Domain

The C backend is strictly split into two threads to guarantee determinism. Network I/O and matching logic must never share a lock, because a lock is a latency spike waiting to happen.

Network Thread
epoll — handles thousands of concurrent TCP connections from AI bots
Reads binary PacketHeader, loops count times, extracts Order structs
Never touches matching logic — pure I/O
SPSC ring buffer
Matching Thread
Pinned to a single CPU core via pthread_setaffinity_np
Polls the lock-free ring buffer continuously
Processes each Order through the Limit Order Book
Subsystem A
Network Thread — I/O

Built on epoll to multiplex thousands of TCP connections from AI market-making bots on a single thread. Its only job is parsing: read the PacketHeader, iterate count times, extract each Order struct, and push it into the ring buffer.

Handoff, not processing. The network thread is deliberately dumb. Any logic placed here — validation beyond struct bounds, business rules, anything — adds latency variance to a thread that exists purely to move bytes.
Subsystem B — The Critical Path
Matching Thread — Execution

Pinned to one CPU core so the OS scheduler never migrates it mid-burst — migration means cold caches, and cold caches mean latency spikes. It polls the ring buffer in a tight loop and processes each order through the Limit Order Book (LOB).

The LOB data structure

Price levels are stored in a hash map (or flat array if the price range is constrained), mapping each price to a PriceLevel struct. Each PriceLevel holds a doubly-linked list of active orders, preserving price-time priority — first in, first out at that price.

malloc is forbidden on this thread

A memory arena is initialized once at startup via mmap. Adding an order bumps a pointer; matching or canceling an order returns its slot to a custom free list. Calling malloc or free in the matching thread reintroduces unpredictable allocator latency — the exact thing the arena exists to eliminate.

03

Market Simulation — Habeeb's Domain

An exchange with no order flow is a demo of an empty room. The liquidity generators exist to make the book look — and behave — like a real market.

Subsystem A
Liquidity Generators

Python or C daemons running locally hold persistent TCP/WebSocket connections to Daniel's network thread, generating valid binary packets in the agreed Order format.

Rather than emitting random noise, the bots run a reinforcement learning policy (Kestrel/RL) that reads the live ExecutionRecord stream as its observation space. The objective: maintain a tight spread — the gap between the best bid and best ask — by continuously placing and canceling thousands of limit orders per second.

This is the same behavior real market-making desks exhibit: constant quote refresh, inventory management, and spread tightening in response to fill pressure. The bots are not decoration — they are the load.

Apex Terminal — Usaamah's Domain

A standard SaaS dashboard cannot survive this workload. 10,000 DOM updates per second will crash Chrome instantly — the terminal must be built like a trading floor, not a web app.

Subsystem A
Render Engine

A Next.js shell wraps a core visualization surface drawn with HTML5 Canvas or WebGL — never the DOM for high-frequency elements. A WebSocket connection receives raw ExecutionRecord arrays as binary frames, parsed directly with DataView and typed arrays (Uint8Array) to extract integers without any JSON deserialization step.
